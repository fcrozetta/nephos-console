# First-run admin setup

- Status: accepted
- Date: 2026-07-13
- Tags: console, auth, setup, security, phase-2

## Context

ADR-0001 chose single-admin session auth with the admin password supplied via
`NEPHOS_CONSOLE_ADMIN_PASSWORD` (an env var on the console). That was a scaffold:
it forces the operator to invent and inject a password out of band, and it lives
only in the console, so it is not the Nephos account model.

The secrets-capability work (nephos `docs/adr/20260713-secrets-capability.md`)
establishes that the **only human-provided credential** is a Nephos admin
account — every service password is generated by Nephos, never supplied by the
user. That admin account must therefore live in Nephos, created on first run,
not baked into a console env var.

## Decisions

### A. Admin lives in Nephos, not the console

The Nephos API owns the admin account (new `admin_accounts` SQLite table, hashed
password). The console authenticates against Nephos; it no longer holds any
password material. `NEPHOS_CONSOLE_ADMIN_PASSWORD` is removed.

### B. First-run `/setup` flow

The console gates on admin existence:

- On any request, the console asks Nephos `GET /auth/state` → `{ adminExists }`.
- If no admin exists, all routes redirect to `/setup`, which collects a username
  + password and calls `POST /admin/accounts`.
- `POST /admin/accounts` succeeds **only when zero admins exist** — the API is
  unauthenticated, so this self-guard prevents account takeover after setup.
- Once an admin exists, `/setup` redirects to `/login`; login calls
  `POST /auth/login` and the console mints its existing HMAC-signed session
  cookie (unchanged from ADR-0001).

### C. Session model unchanged

The server-side, HMAC-signed cookie session from ADR-0001 stays. Only the
credential *source* changes: from an env var to a Nephos-verified login. The
OIDC seam (Zitadel) remains the future replacement for the whole login step.

## Consequences

- A fresh install has no password anywhere until the operator sets one through
  the browser — no secret to inject, no default to leak.
- The console depends on three new synchronous (non-202) Nephos endpoints
  (`GET /auth/state`, `POST /admin/accounts`, `POST /auth/login`); these are
  auth/identity, not reconciled desired state, so they are correctly synchronous.
- Until those endpoints exist in Nephos, the console keeps the ADR-0001 env
  password as a stopgap; the switch lands with the Nephos-side change.

## Follow-up

- Multi-admin / roles (out of scope; single admin only).
- OIDC login via Zitadel replacing password login.
- Session revocation / rotation on password change.
